Teoría

Que es git?

Git es un software de control de versiones muy conocido y el más utilizado del mundo. Su creador, Linus Torvalds también conocido como el creador del kernel del sistema operativo Linux, su idea principal era la de un motor de bajo nivel sobre el cual otros pudieran escribir la interfaz de usuario, no obstante git con el tiempo se ha convertido en un sistema de control de versiones con funcionalidad plena.

Como funciona git?

Para entender como funciona git tenemos que tener claro el concepto de ramas. Una rama podríamos describirla como un histórico de cambios realizados sobre un código especifico. Podemos distribuir estas ramas como queramos para tener un mejor control y orden de nuestro código, a esto se les llaman metodologías como por ejemplo Git Flow. Git nos permite recuperar el estado anterior de nuestro código para solucionar un error o devolver el código al punto en el que funcionaba bien.

Git tiene muchas ventajas

  • Desarrollo simultáneo
  • Muy potente
  • Versiones más rápidas
  • Software Libre
  • Soporte técnico solido
  • Funciona con cualquier equipo

    • Que es la metodología GitFlow?

    Git Flow es un modelimg/35.pngoativo de creación de ramas en Git en el que se utilizan ramas de función y varias ramas principales. Su creador fue Vincent Driessen.

    Es una estrategia creada para mejorar la organización de las ramas dentro del repositorio y dar más fluidez al proceso de nuevos Features y Releases

    Gitflow presenta una solución más robusta que git, sobretodo para grandes proyectos o proyectos que requieren actualizaciones constantes. También se utiliza cuando existe una gran cantidad de personas trabajando dentro de un repositorio.

    Como funciona?

    Hay dos tipos de ramas dentro de Git Flow:

    1. Principales: Master y Develop
    2. Soporte: Feature, Release y Hotfix

    Para cada rama hay existen unas reglas específicas:

    • Master/Main
      • Cualquier commit que pongamos en esta rama debe estar preparado para subir a producción
    • Develop
      • Rama en la que está el código que conformará la siguiente versión planificada del proyecto
    • Feature
      • Nuevas funcionalidades de la aplicación
    • Release
      • Se realizan los últimos ajustes y se corrigen los últimos bugs antes de pasar el código a producción
    • Hotfix
      • Se corrigen los errores y bugs que hayan en producción

    Crear repositorio

    Para crear un repositorio ejecutamos el comando

    git init

    Esto generará la carpeta oculta .git y creara el nombre de la rama que tengamos asignada por defecto, en mi caso main

    (git config --global init.defaultBranch main para modificar el nombre de la rama por defecto a main)

    Ahora crearemos un fichero README.md con la descripción de nuestro proyecto y haremos el primer commit

    Luego tendremos que crear un repositorio en github, en mi caso lo he llamado Practica-GitFlow

    El siguiente paso sera enlazar nuestro repositorio local con el remoto de github, para ello utilizaremos el comando

    git remote add origin -link-

    El -link- es la url que nos ofrece github cuando creamos el repositorio en mi caso:

    https://github.com/JoaquimRS/Practica-GitFlow.git

    Ahora publicamos nuestra rama local main al repositorio remoto con el comando

    git push origin main

    Seguidamente nos pedirá el token de nuestra cuenta de github

    (https://docs.github.com/en/enterprise-server@3.4/authentication/keeping-your-account-and-datasecure/ creating-a-personal-access-token para crear un token personal en github)

    Ahora introducimos los usuarios que van a colaborar en nuestro proyecto. En mi caso ximo-ribera, Donyet960

    Git nos enviara un correo para aceptar la invitación, mientras tanto los usuarios estaran pendientes de invitación.

    Para poder publicar los cambios con diferentes usuarios tendremos que hacer lo siguiente

    Borrar el origin

    git remote remove origin

    Y ahora añadir a todos los usuarios con el nombre del origin con su nombre de usuario

    git remote add JoaquimRS https://github.com/JoaquimRS/Practica-GitFlow.git

    git remote add ximo-ribera https://ximo-ribera@github.com/JoaquimRS/Practica-GitFlow.git

    git remote add Donyet960 https://Donyet960@github.com/JoaquimRS/Practica-GitFlow.git

    Así cada vez que hagamos el push tendremos que poner el nombre del usuario como origin para diferenciarlo, y seguidamente pedirá el token de cada usuario respectivamente

    Metodología Git Flow

    Ahora vamos a crear la metodología GitFlow

    Para ello crearemos otra rama llamada develop a partir de nuestra rama principal main

    Para cada mejora crearemos una feature a partir de la rama develop y cuando terminemos de hacer la mejora hacemos un merge con la rama develop para así poder implementarla

    Cuando queramos publicar una versión haremos una release sobre la rama develop para asi poder hacer un merge tanto con la rama main como con develop con las mejoras que queramos añadir a la versión

    Y finalmente si queremos arreglar un error que tengamos en la rama main crearemos la rama hotfix para arreglar los errores que haya en la rama principal y aplicaremos los cambios haciendo merge tanto en main como en develop para así arreglarlo de forma correcta en las dos ramas

    Para luego poder ver los cambios que hemos hecho y la estructura publicaremos la rama develop en el repositorio remoto

    También vamos a crear la rama gh-pages para así poder tener nuestra pagina html del repositorio visible y dentro pondremos la explicación de nuestro proyecto

    Release

    Para crear la release hay que situarse en la rama develop y crear la rama release_v1.0

    Con esta release lo que haremos sera publicar todas la features que hemos hecho y ponerle una etiqueta para deferenciar entre versiones, luego todo esto lo uniremos a la rama main

    git push Donyet960 release_v1.0 (Publicar la rama)

    Ahora toca asignar la etiqueta v1.0 al merge que hemos hecho con la release en la rama main

    git checkout main (Cambiamos a la rama main)
    git pull Donyet960 main (Actualizamos la rama main con el repositorio remoto)
    git tag v1.0 dce4b2 (Asignamos la etiquet al merge que hemos hecho)
    git push --tags Donyet960 (Publicamos la etiqueta)
